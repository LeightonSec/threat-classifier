## 1. Training Data Policy

### 1.1 Positive class (`threat`) examples

Positive training examples must be **synthetic** — generated by llm-redteam tooling or manually crafted for training purposes. They must not be sourced from:

- `honeypot.db` — verbatim attacker prompts, classified CRITICAL sensitivity. Ingesting these at training-data volume introduces data-at-risk for the duration of the training artefact's lifecycle, which is undefined and likely longer than the 90-day `honeypot.db` retention window. The training dataset would become a second, unmanaged copy of the honeypot corpus.
- Live `pcap-analyser` captures — src IPs, MACs, DNS query content are HIGH sensitivity. Including raw network captures in training data creates an indefinite retention of HIGH sensitivity data in an artefact that may propagate outside the operator's control boundary.
- Live `ai-firewall` outputs — verbatim prompts from `ai-firewall` are HIGH sensitivity and already represent a pre-existing boundary violation (documented in Gate 2). Using them as training data compounds the violation.

**Rationale for synthetic-only rule:** llm-redteam was built to generate attack payloads representative of real techniques without requiring live attacker data. The trade-off is some distribution mismatch between synthetic payloads and real attacker behaviour — this is accepted. The alternative (using live attacker data from `honeypot.db`) introduces data handling obligations that are not manageable within the current TC architecture.

**Distribution mismatch acknowledgement:** synthetic payloads may not cover novel attacker techniques or obfuscation strategies not present in llm-redteam's generation templates. This is residual risk accepted at Gate 4. Mitigation path: extend llm-redteam templates to cover new evasion patterns as they are observed; do not relax the live-data prohibition to compensate.

### 1.2 Negative class (`benign`) examples

Negative class examples may be sourced from public datasets of non-malicious text (log lines, network metadata, legitimate prompts). Examples must be confirmed non-malicious and must not include PII or sensitive operational data.

### 1.3 Training dataset versioning and access control

The training dataset must be:
- Version controlled separately from the public threat-classifier repo.
- Not committed to the public repo, even if individual examples appear benign — the full dataset reveals coverage and blind spots of the classifier.
- Documented by source category (synthetic payloads / public benign corpus) in the training script.

## 2. Model Weights Storage

### 2.1 Storage requirement

Fine-tuned model weights must not be committed to the public threat-classifier repo. The threat model for public weight exposure is documented in Gate 3 Section 4.4 (white-box attacks): gradient-based adversarial examples can be constructed efficiently from public weights.

**Allowed storage locations:**
- Local filesystem — accessed via `MODEL_PATH` environment variable
- Private repository
- Object storage with access controls (e.g., S3 bucket with IAM policy, no public ACL)

**Prohibited:**
- Committed to `LeightonSec/threat-classifier` or any public fork
- Published to Hugging Face Hub under a public model card without explicit sign-off at a future gate

### 2.2 Loading mechanism

The inference pipeline must load weights via `MODEL_PATH`:

```python
import os
from pathlib import Path

model_path = Path(os.environ["MODEL_PATH"])
```

`MODEL_PATH` must have no default value in code — if unset, the inference pipeline must fail loudly rather than silently falling back to a stub or pulling weights from a remote source. This prevents accidental production deployment of the Phase 1 keyword stub.

### 2.3 Training script documentation

The training script must document:
- Dataset source categories consumed
- HuggingFace model ID used as base (with pinned `revision=` commit hash — consistent with Gate 3 model selection and the ai_ml scanner check)
- Output path convention for saving weights

This satisfies the auditability requirement for the model weights checkpoint without requiring the weights themselves to be in the public repo.

## 3. TC → Anthropic API Payload Enforcement

Carried from Gate 2 (Flow 2 sign-off) and Gate 3 (Section 7).

### 3.1 Requirement

The PR that introduces the Anthropic API call in threat-classifier must be reviewed against the following payload constraint:

The `content`/`messages` payload passed to the Anthropic API must contain **only classifier output**:
- Binary label: `threat` or `benign`
- Confidence score: float in `[0, 1]`

It must **not** contain:
- Raw alert text from any signal source
- Any attacker-supplied string from `llm-honeypot`, `ai-firewall`, `llm-redteam`, or `pcap-analyser`
- Intermediate normalised text (even NFKC-normalised attacker content is still attacker-controlled)
- Any field that has not been reduced to a score or label by the DistilBERT classifier

### 3.2 Enforcement mechanism

This is a **PR review checkpoint**, not a static analysis check. The security-gate scanners verify that existing code does not send raw strings to the Anthropic API, but they cannot verify that the classifier output is the *only* content in a new API call until the call is written.

Reviewer checklist item for the Phase 2 Anthropic API PR:
- [ ] `content` field contains label + score only
- [ ] No f-string or concatenation of raw signal fields into the prompt
- [ ] No pass-through of `.alert_text`, `.raw_prompt`, `.payload`, or equivalent fields from signal source models

### 3.3 Relationship to pre-existing violations

The pre-existing `ai-firewall/detector.py:api_scan()` violation (verbatim prompts sent to Anthropic API) is documented in Gate 2 and is not TC's responsibility to fix. The TC → Anthropic API flow must be clean regardless of what upstream components do — TC must not ingest ai-firewall output and forward it to the Anthropic API.

| Condition | Status |
